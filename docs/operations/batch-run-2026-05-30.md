## Agreed execution spec (reusable for future batches)

**Unit of work:** 1 issue → 1 worktree → 1 branch → 1 PR.

**Per-issue loop (author subagent):**
1. Implement on its own worktree/branch (`claude/issue-<n>-<slug>`).
2. Objective gate — tests + linters for touched components must pass
   (`go test`/`go vet`, `biome`, `ruff`; `cross-sdk-tests` for receipt-format changes).
3. Self-run `/code-review` to burn down obvious findings.
4. Open PR, request Copilot review.
5. Address findings — **mechanical/reasonable → auto-fix; anything ambiguous or
   judgment-laden → HALT and ask the user.**
6. Re-request Copilot review. Repeat until Copilot leaves no new suggestions,
   or **3 fix rounds** (hard ceiling 5).
7. On cap-without-convergence → stop, leave PR open, summarize unresolved findings, escalate.
8. **Never merge.** Deliverable = a clean PR for the user to review + merge.

**Serial/dependent issues** → stacked PRs (each branched off the prior issue's
branch); re-rebase downstream if an earlier PR changes during cleanup.

**Wave gating:** prep + clean one wave, then **HALT** and wait for explicit "go".
User merges between waves. Human merge gate always — doubly so for CI
(`.github/workflows/`), `spec/`, and crypto.

**"Copilot-clean" is NOT a stable terminal state (observed on #678):** after the c8ed965 review
returned "no new comments" (which I reported as clean), Copilot ran *again* and produced 8 then 2
*new* substantive accuracy comments on the same files. So a single "no new comments" pass is not a
guarantee the reviewer is done — a later/deeper pass can surface more. *Future tuning:* before
declaring a PR clean, either wait for a quiescent period with no pending Copilot run, or treat "clean"
as provisional until N minutes pass with no new review. Don't report "clean → ready to merge" off one
green pass for substantive (non-trivial) PRs.

**Copilot fix-loop convergence (observed on #679):** Copilot surfaces NEW incremental items each round
(R1: 2 → R2: 3 → R3: 4 comments), severity decreasing (real bug → robustness → consistency + docs).
Every pass was legitimate, but a thorough reviewer rarely returns "zero" quickly on a non-trivial
feature PR. This is why the 3-round cap + halt-to-human exists. *Future tuning:* for feature PRs (vs
docs), front-load a stricter self-review / `/code-review` pass before the first Copilot request to
collapse rounds; treat "new lower-severity nits each round" as the signal to stop and let the human
take the last mile.

## Improvements / retro (fill as we go)

**Environment / setup friction (wave 1 startup):**
- Fresh container had **no `origin/HEAD`** set → the `test-before-push.sh` hook fell
  back to running ALL four test suites on every push. Fixed with
  `git remote set-head origin main`. *Future fix:* a SessionStart hook should set this.
- **sdk/ts** had no `node_modules` and **sdk/py** had no usable venv → push hook failed
  on unrelated suites. Fixed with `pnpm install` (sdk/ts) and `uv sync --all-extras`
  (sdk/py — note: plain `uv sync` is NOT enough; pytest lives in the `dev` extra, and
  without it `uv run pytest` silently picks up a global pytest that can't import the
  package). *Future fix:* SessionStart hook should run these installs. The
  `session-start-hook` skill is built for exactly this.
- The push hook runs the **full matrix when a branch diffs from main on multiple
  components** (this orchestration branch carries prior SDK commits #669–671). Per-issue
  branches forked from `main` only trigger the suites for paths they touch, so this is a
  non-issue for the per-issue PRs — but it makes pushes to THIS branch slow (~full suite).

**Worktree-isolation race (BIG one — affects how to launch future waves):**
- Launching 3 `isolation: "worktree"` agents in one batch raced on the shared repo HEAD.
  All three got their own worktrees and ultimately pushed correct, scoped PRs (#677/#678/#679),
  BUT the setup churn left the **main worktree** parked on a stray `issue-536` branch, with my
  tracking commit misplaced onto it and leftover `daemon/go.mod` cruft (#441's `BurntSushi/toml`
  edit captured while main HEAD transiently sat on the issue-441 branch). No data lost; origin/main
  never touched; recovered by switching back to the orchestration branch, re-applying the doc, and
  deleting the stray branch.
- **Lesson:** don't trust the shared HEAD while parallel worktree agents spin up. Either (a) stage
  the orchestrator on a throwaway branch and don't commit/push from main while agents launch, or
  (b) launch author agents one-at-a-time (serial waves already do this), or (c) verify
  `git status` + branch before every orchestrator commit. Per-issue isolation itself worked —
  the hazard is the orchestrator's own working tree.
- **It recurs on EVERY parallel agent launch:** the address-round launch churned main onto
  `claude/issue-441-daemon-config` with staged residue again, and the new worktrees even contained
  an incomplete uncommitted copy of a fix. Confirmed harmless each time (real output lands on origin),
  but the standing mitigation is: after launching any worktree agents, treat main's branch/index as
  untrusted — `git checkout -f <orchestration-branch>` to reset before doing orchestrator commits.
  Also a known stop-hook trigger (uncommitted residue in main).
- Each author worktree also saw the untracked `docs/operations/batch-run-2026-05-30.md`; agents
  correctly left it unstaged. Author prompts should still say "ignore untracked orchestration files".
